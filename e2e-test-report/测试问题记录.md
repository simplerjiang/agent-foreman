## 问题清单

### 问题 #1 — 重复启动会开出第二个窗口（非单实例）🟡 Minor
- **位置**：应用启动 / 进程管理
- **现象**：`foreman.exe` 已在运行（占用 `127.0.0.1:8788`）时，再次启动 `foreman.exe`，不会聚焦已有窗口，而是**又弹出一个新的窗口**。新实例检测到端口被占用后，没有自己起服务，而是直接加载已有后端（这一点处理得不错），但结果是用户桌面上出现**两个一模一样、都连着同一个后端**的 Foreman 窗口。
- **证据**：测试时进程列表里出现两套 foreman 进程（PID 18304 @12:46 起的服务端 + PID 9924 @12:52 的重复窗口），但只有 18304 在监听 8788 端口。
- **影响**：桌面级工具如果允许开多个相同窗口，用户容易搞混"我在哪个窗口操作"，且任务栏出现多个图标。成熟的桌面应用（VS Code、Slack 等）默认单实例——再次点击图标会把已有窗口拉到前台。
- **建议**：做成单实例应用。第二次启动时检测到端口/锁文件已存在，直接把已有窗口前置（focus + restore），而不是新开窗口。这也契合"极简"原则——一个工具就一个窗口。

### 问题 #2 — 中文界面里 PM 流程状态全是英文（语言混杂）🟠 Major
- **位置**：工作台 → 会话时间线（PM 状态条 / 计划卡）
- **截图**：`screenshots/02-语言混杂-PM状态英文.png`
- **现象**：用户处于「个人模式 · 中文界面」，左侧导航、按钮全是中文，但会话时间线里 PM 的关键过程提示却是英文：
  - `PM is planning agent choice, todo list, and launch instruction…`
  - 计划卡正文：`PM tool loop reached max rounds before a final plan; dispatching a conservative fallback instruction.`
  - `PM tool loop hit max rounds; using fallback plan.`
  - `PM selected claude-code; launching the coding agent…`
  - claude-code 回复：`0 cmd · I'm ready to help. What would you like me to work`
- **影响**：这些恰恰是用户最需要看懂的"工作进程"信息（PM 在干嘛、选了哪个 agent、为什么用了 fallback）。一个中文用户读到一屏英文技术黑话，完全无法理解 Foreman 此刻的状态，直接违背"让用户更清晰直观了解工作进程"的目标。
- **建议**：所有面向用户的 PM 过程状态/计划文案都要走 i18n，跟随界面语言。状态条用人话：例如「PM 正在挑选 agent、拆解步骤…」「PM 选了 claude-code，正在启动…」。

### 问题 #3 — "PM tool loop hit max rounds; using fallback plan" 暴露给用户且无人话解释 🟠 Major
- **位置**：工作台 → 会话时间线 → PM 计划卡
- **截图**：`screenshots/02-语言混杂-PM状态英文.png`
- **现象**：仅仅是一句"早上好，本地agent情况如何"的问候，PM 大脑就"工具循环达到最大轮次、没产出最终计划、改用保守兜底指令"。这串文案直接抛给用户，既是英文，又是内部实现细节（tool loop / max rounds / fallback plan）。
- **影响**：① 体验上：用户看不懂，也会担心"是不是出错了"；② 这是 PM 规划环节本身的效率问题（过度规划 / 轮次上限过低 / 工具循环不收敛）。
- 📌 **gpt-5.5 复测结论（已确认，见 #20）**：原以为"换成 gpt-5.5 后就不会再这么轻易兜底"——**复测证明并非如此**：即便 PM 大脑模型是正常工作的 `gpt-5.5`，对"用一句话说一下 agent 是否就绪"这种极简单请求，PM **仍然打满 max rounds（6）→ 用降级计划**（计划卡 "0 步" + 英文 fallback 文案）。所以 **#3 与 #6/#9 的 502 无关，是独立的真 bug**。
- **建议**：① 兜底是内部机制，不该作为"计划"正文展示；若要展示，用中文人话且主文案要 i18n（现仅副标题是中文"PM 工具循环已达到轮次上限：将使用降级计划"）。② **重点修复**：排查 PM 规划为何对简单请求也打满轮次——让简单请求短路跳过繁重规划，或修工具循环不收敛；配合 #19 把轮次上限做成可配置。

### 问题 #4 — 摘要/标题取了模型的"英文思考开场白"，而非真正答案 🟡 Minor
- **位置**：工作台 → 会话卡片折叠摘要 / 子代理卡片标题 / 终端
- **现象**：claude-code 的真实回复其实是中文的（"你好！当前本地 Agent 状态：…"），但在三个地方展示的摘要/标题都取了回复的第一行——也就是模型把思考过程泄漏到正文的英文开场白 `I'm ready to help. What would you like me to work on?` / `The user wants me to respond in Simplified Chinese…`。
- **影响**：用户在折叠态、子代理列表里看到的都是一句无意义的英文，看不到 agent 实际产出的中文结果，降低"一眼看懂进展"的效率。
- **建议**：摘要应跳过明显的思考/开场白（"I'm ready to help"、"The user wants me to…"这类），优先展示实质结论；或直接取末段。三处摘要逻辑应统一。

### 问题 #5 — "终端"实为只读输出日志，且 Windows 上用了 mac 红黄绿灯 🔵 建议
- **位置**：工作台 → 右栏 → 终端 tab
- **现象**：① "终端"tab 实际是 agent 的只读原始输出日志（markdown 的 `**` 没渲染、纯文本堆叠），并不是可输入的真终端；命名容易让人以为能在里面敲命令。② 面板顶部用的是 macOS 的红/黄/绿三个交通灯圆点，但这是 Windows 桌面应用，风格不一致。
- **建议**：① 若不是交互终端，可考虑改名为"原始输出/日志"，或明确这是只读；② 去掉 mac 交通灯，或换成与 Windows 风格一致的极简标题栏。

### 问题 #6 — LLM 调用失败时，两处错误处理"一好一差"（错误处理不一致）🟠 Major（错误处理不一致）
> 📌 **更正（依据用户反馈）**：原以为是"PM 大脑端点 `api.kongsites.com` 宕机/502"。**真正根因是模型配成了 `gpt-4o`，而服务商没有该模型，所以返回 502**——这属于测试配置问题（应选 `gpt-5.5`，见上方「测试前置条件」），**不是产品 bug**。本条**仅保留"错误处理风格不统一"这一真正有效的发现**。
- **位置**：简报 → 生成简报；以及工作台 → 会话规划
- **截图**：`screenshots/06-简报生成失败-502.png`
- **现象**：当 LLM 调用失败时（本次因模型 `gpt-4o` 不存在触发 502），不同入口的报错质量差别很大：
  - ✅ **简报页**：错误处理做得很好——中文标题"PM简报生成失败"、给出原始错误、并明确指引"请检查 ① `.env` 里的 `FOREMAN_LLM_API_KEY` ② 「PM 大脑」设置页的 服务商/模型/接口地址"，还提供「复制」按钮。这是优秀范例。
  - ❌ **会话规划**：同样是 LLM 调不通，却被吞成一句英文"PM tool loop hit max rounds; using fallback plan"，既没说真正原因，也没指引去哪改配置。用户完全被误导（与 #3 同源）。
- **影响**：错误处理风格不统一、体验割裂；尤其会话规划这条主链路在失败时几乎不给有效信息。
- **建议**：统一错误处理——会话规划失败时也应像简报页一样，明确告诉用户失败原因（如"模型不存在 / 调用失败"）并给出修复指引（去 PM 大脑设置改模型为 `gpt-5.5`），而不是显示内部的 max-rounds 兜底术语。详见汇总 #17。

> ⚠️ **测试环境说明（已更正）**：本轮因 PM 大脑模型配成了不存在的 `gpt-4o`（应为 `gpt-5.5`，见「测试前置条件」），"PM 规划 / agent 下发 / 简报生成 / 压缩上下文"等依赖 LLM 的链路**没能跑通**——这是测试配置所致，**非产品限制**。因此本轮重点落在：界面/交互/可用性、本地不依赖 LLM 的功能、各类错误与边界状态、产品合理性审视。**LLM 相关链路（含决策卡/审批流）需在模型改为 `gpt-5.5` 后重新回归。**

### 问题 #7 — 工作方式「适用范围(JSON)」填非法 JSON 时，保存静默失败、无任何提示 🟠 Major
- **位置**：工作方式 → 新建/编辑 → 适用范围 (JSON)
- **截图**：`screenshots/07-工作方式-非法JSON静默失败.png`
- **现象**：在「适用范围 (JSON)」里填入非法 JSON（如 `{这不是合法的JSON,,,`），名称、内容都填好，点击「保存」——弹窗**不关闭、不保存、也不报任何错**。反复点击都没有反应，用户完全不知道是 JSON 写错了。
- **对比**：合法 JSON（`{}`）时保存正常（已验证 happy path 可用）。所以是 JSON 解析失败被静默拦截了。
- **影响**：用户会以为"保存按钮坏了"或"程序卡死"，无从下手。这个字段还要求用户手写 JSON，本身门槛就高，再加上没有报错提示，非常劝退。
- **建议**：① 解析失败时，在字段下方给出红字提示（如"适用范围不是合法 JSON，请检查"）并高亮该字段。② 更进一步（更极简）：把「适用范围」做成可视化选择（语言多选 + 路径 glob 输入），避免让用户手写 JSON；保留"高级：编辑 JSON"作为折叠选项。③ 字段旁给一个示例或占位符（如 `{"languages":["python"],"paths":["**/*.py"]}`）。

### 问题 #8 — 主题/语言/手机通知开关：位置不统一 + 重复 + 最常用的工作台反而没有 🟡 Minor（违背极简）
- **位置**：各页右上角浮动按钮（🌞主题 / EN语言 / 🔔开启）与「设置」页内的对应行
- **现象**：① 这组全局开关在 决策 / 简报 / 工作方式 / 设置 四个页的右上角都有浮动按钮；② 但在**最常用的「工作台」页右上角却没有**（那里被"自动权限 dial + 生成简报"占据），导致用户在工作台时无法切主题/语言；③ 同时这三项在「设置 → 界面与自动化」里又各有一行（主题 浅色/深色、语言 中文/EN、手机通知 开启）——**同一组控制散落两处、且行为重复**。
- **影响**：违背"极简"。用户找一个开关要看运气在哪个页；相同功能两套入口增加认知负担。
- **建议**：全局偏好（主题/语言/通知）收敛到**一个稳定位置**——要么统一放在所有页都可见的固定顶栏角落（含工作台），要么只放在「设置」里，不要两头都有、还挑页出现。

### 问题 #9 — LLM 调用失败时，新会话永久卡在"运行中 / PM 正在规划…"，无失败态、无人话、无重试 🔴 Blocker（失败健壮性）
> 📌 **更正（依据用户反馈）**：触发本条的 502 根因是模型配错（`gpt-4o` 不存在，应为 `gpt-5.5`），**那次"连不上"本身不是 bug**。但本条要说的不是"连不上"，而是**"一旦 LLM 调用失败，会话就永久假装在跑、无法收尾"**——这对任何失败场景（选错模型、限流、超时、网络抖动）都成立，属于**失败健壮性**问题，保留。
> ✅ **gpt-5.5 已复测确认（第 5 轮，复现 3 次）**：换成正常的 `gpt-5.5` 后，端点偶发 **408 Request Timeout** 时，会话**照样永久卡在"运行中 / PM 正在规划…"**，无重试、无失败态。所以这是与模型无关的真·健壮性 bug，且现实里就会发生（端点抖动）。另注：本轮失败事件标签是「错误」，而原始 502 那次标的是「摘要」——**同类错误事件的标签也不一致**，需统一。
- **位置**：工作台 → 下发新任务（实测下发了一条只读测试指令）
- **截图**：`screenshots/09-会话规划卡死-502无失败态.png`
- **现象**：下发任务后，PM 调 LLM 失败（本次因模型 `gpt-4o` 不存在返回 502），但会话**没有进入失败态**——等待 25 秒以上，会话头部依旧显示"● 运行中"，左侧会话列表依旧"pm-agent · running"，正文一直转圈"PM 正在规划 agent 选择，任务清单和执行指令…"。唯一的线索是时间线里一条灰色系统事件：`摘要 · HTTPStatusError: Server error '502 Bad Gateway' …`。
- **关键问题**：
  1. **假"运行中"**：会话其实已经死了，但 UI 永远显示"运行中"，用户会一直干等，不知道它不会再有进展。
  2. **没有人话错误**：错误以英文原始堆栈出现在系统事件里，标签还是"摘要"（应为"错误/失败"）。对比 #6 的简报页——同样的 502，简报页给了清晰中文失败卡 + 修复指引，会话流却完全没有。两条链路对 LLM 失败的处理天差地别。
  3. **无重试/无收尾**：没有"重试"按钮，也没有把会话标记为 failed，用户只能自己去左侧删/新建。
- **影响**：这是**核心链路**（下发任务 = 产品主功能）。一旦 PM 端点抖动/超时，用户得到的是一个"看起来在跑、其实卡死"的会话，体验和信任都会崩。即便 LLM 偶发 502 也应优雅降级。
- **建议**：① PM 调用失败要让会话进入明确的**失败态**（头部状态变"失败"，停止转圈）；② 用 #6 简报页那套中文失败提示（"PM 大脑调用失败：502，请检查接口地址/Key"）+ 一键**重试**按钮；③ 给 PM 调用加超时与有限重试，超时即收尾，不要无限挂起；④ 系统事件标签"摘要"用错了，错误事件应标"错误/失败"。

### 问题 #10 — 会话无法停止 / 取消 / 删除（agent 控制器缺最关键的"刹车"）🔴 Blocker（产品定位）
- **位置**：工作台 → 会话（头部 / 侧边栏）
- **现象**：找遍了都没有"停止/取消/删除会话"的入口——会话头部只有标题 + "运行中"徽标 + 自动权限 dial + 生成简报，**没有停止按钮**；右键侧边栏会话**没有上下文菜单**；鼠标悬停会话项**也不出现删除/停止图标**。当前三个会话全部卡在"running"，用户没有任何办法把它们停下来或清理掉。
- **影响**：对一个"桌面级 + 远程 agent 控制器"来说，**能停下正在跑的 agent 是最核心的安全控制**。现在：① agent 跑飞了/卡死了（如 #9）用户无法叫停；② 历史会话越积越多且全是"running"，无法删除归档，列表会越来越脏；③ 远程场景下更危险——人在手机端看到 agent 在乱来，却没有"急停"。
- **建议**：① 会话头部加醒目的**「停止」**按钮（运行中可见），点了能真正终止底层 agent 进程；② 侧边栏会话支持悬停出现 ⋯ 菜单或删除图标（停止 / 重命名 / 删除 / 归档）；③ 远程控制器尤其要有"一键急停所有 agent"的全局开关。这是该品类的刚需，不是锦上添花。

### 问题 #22 — 顶部"PM 正在规划…"状态条 spinner 永不消失 🟠 Major
- **位置**：工作台 → 会话时间线顶部的 PM 状态条
- **截图**：`screenshots/21-gpt5.5-完成但仍运行中+规划spinner不消失.png`
- **现象**：会话刚下发时出现"↻ PM 正在规划 agent 选择、任务清单和执行指令…"的转圈状态条。但**直到计划已出、agent 已跑完、PM 已复查通过，这条"正在规划…"仍在原地转圈**，从不清除。
- **影响**：这是一个**会撒谎的进度指示**——它一直说"正在规划"，用户会以为 PM 卡在规划阶段，实际上早就规划完、任务都做完了。直接违背"让用户清晰直观了解工作进程"。
- **建议**：状态条应随阶段推进实时更新/清除：进入下一阶段就替换成对应文案（"正在执行…""复查中…"），全部结束就收起，不要让"正在规划…"一直挂着转。
