💬 **二次修复**：#39 被 E2E 打回过，按规约不许「裸关甩异步」，必须当场点过、附实测证据才能 close。

## 1. 前置条件（必须先做对，否则全部用例会假失败）

| 项 | 要求 | 说明 |
|---|---|---|
| **被测物** | 本轮**新打包的 `dist/foreman.exe`**（main 已合 PR #100） | 不要用旧 exe；从源码跑亦可（`foreman app`，加载 `.env` 的 LLM key） |
| **PM 大脑模型** | 设置里设为 **`gpt-5.5`** | ⚠️ **关键**：填 `gpt-4o` 之类不存在的模型会让所有 LLM 流真失败，污染用例。正向用例一律用 `gpt-5.5` |
| **传输 / 后端** | `provider=openai`、`base_url=https://api.kongsites.com/v1`、`transport=ws` | ws 端是用户自建 CLIProxyAPI，桥接 Codex 订阅；**别动它**（基调 §0.5-7「传输不动地基」） |
| **密钥** | 在 `dist/.env`，**严禁打印 / 截图 / 回显**到报告 | 报告里出现 key 即作废 |
| **坐标** | computer-use 坐标 **1:1**，点击用原子 PowerShell 单击 | 见 foreman-e2e skill（避免坐标缩放 / 连点漂移） |
| **干净库** | 准备一个**可被 PM 审阅的真实小仓库/工作区** | UC-1 要让 PM 真的跑「审阅代码」并出计划 |

> 设置入口：app 内「设置 / Settings」→ PM 大脑（model / base_url / transport）。改完重进一个新会话再测。

## 2. 验收用例（逐条点、逐条截图、逐条勾）

> 每条用例：**操作 → 期望 → 截图点 → 判定**。截图命名建议 `UC<n>-<关键帧>.png`，落 `e2e-test-report/`。

### UC-1 ▶ 正常「审阅代码」任务：plan 秒级收口、只产一次 `submit_plan`、正常 dispatch
对应 §7 **A/B/F** + **P1 验收（合）**。这是根因是否真治好的主判据。

1. 新建会话，发一条真实的「审阅这个仓库的代码并给出计划」类任务（指向前置准备的小仓库）。
2. 观察规划阶段 → 计划产出 → 派发 subagent 全过程。

**期望（全部成立才算过）：**
- [ ] 规划在**秒级 ~十几秒内收口**，**不卡死、不空转**（不再是几十秒~分钟级的假 running）。
- [ ] 产出的是**一份真实计划**（有 summary / agent / instruction / todo），**不是**空的 conservative fallback（"PM tool loop reached max rounds…"）。
- [ ] 计划**只出现一次**，**没有**同一份计划被复读 / 重复派生的痕迹。
- [ ] 计划落定后**正常 dispatch** 出 subagent，能看到工具调用反馈。

📷 截图点：规划中（有进度）、计划产出瞬间、dispatch 后的 subagent。

> 💬 为什么这条是核心：A1 强制工具让「提交轮」只能产一次 `submit_plan` 工具调用，turn 收到一个 tool_use 即终结——协议层就不可能复读 47 次。

### UC-2 ▶ 失败/停滞**可见**：会话落 `failed`/`stalled` 终态 + UI 显示 reason，绝不假 `running`
对应 §7 **C** + **P0 验收（合）②**。验「失败要响、要可见」（基调 §0.5-4）。

1. 临时把 PM 大脑模型改成一个**不存在/会失败的模型**（如 `gpt-4o`），制造一次 LLM 失败。
2. 新建会话发任意规划任务，观察会话状态与界面。

**期望：**
- [ ] 会话**不再永久 `running`**；落到**明确终态**：界面出现 `failed` / `stalled`（红色失败态）。
- [ ] 界面/错误节点**展示失败原因 reason**（如墙钟超时 / 无进展 / 复读 / stalled 的人话文案），不是空白。
- [ ] 失败态会话提供**重试**入口（retry 覆盖 `failed` 和 `stalled`）。

3. **测毕把模型改回 `gpt-5.5`**（别留坏配置污染后续用例）。

📷 截图点：失败态卡片（含红标 + reason 文案）、重试按钮。

> 💬 这条同时验掉了「老 bug 的最坏观感」：以前 LLM 一挂就假装在跑，现在必须亮红 + 说原因。

### UC-3 ▶ 规划期 UX「有生命」：计时 + todo 渐显 + 工具反馈
对应 §7 **E**（T2.1 / T2.2）。

1. 在 UC-1 的正常任务里，盯住规划阶段那段时间。

**期望：**
- [ ] `pm-status` 节点有**「已 N 秒」自走秒表**，每秒在动——即使没有新内容，也不像冻屏白屏。
- [ ] 计划的 **todo 是逐条渐显**入场（0→N 增长时一条条出现），不是「啪」一下整块塞出来。
- [ ] 正常任务能看到 **subagent 启动**与**工具调用（tool_pre/tool_post）反馈**。

📷 截图点：规划中「已 N 秒」、todo 渐显中间帧、工具调用反馈。

### UC-4 ▶ Stop **真取消**：点取消 → ws 实际断、会话立即停
对应 §7 **E** + **T2.3**（codex 复验重点）。

1. 起一个正在规划/运行的会话（live 态），点 workspace 头部「取消会话 / Stop」。

**期望：**
- [ ] 点击后会话**立即停**（落 `cancelled`），**不是**只改个标签、底层还在偷偷跑。
- [ ] 不会再冒出该会话的后续规划/派发事件（ws 调用确实被取消）。

📷 截图点：取消前（live）、取消后（cancelled，无后续事件）。

> 💬 验证点：`cancel()` 先标 `cancelled` 再真正 cancel 该会话的 launch task，`CancelledError` 经 `ws.recv()` 让 `async with ws` 退出即关 socket。

### UC-5 ▶ 跨会话 todo **不串台**（codex Low 项回归）
对应 codex 第三轮 Low：`TodoPanel` 按 session key 重挂载。

1. 会话 A 跑出一批 todo（已渐显完）。切到会话 B（也有 todo），再切回 A。

**期望：**
- [ ] 切换会话后**不会**从上一个会话的进度继续「重放」历史 todo 动画；每个会话显示自己的 todo，满载会话直接全显、不重播。

📷 截图点：A 的 todo、切到 B 的 todo（各自正确，无错位重放）。

### UC-6 ▶ review / compact 路径同样不挂死（「一处修复四处覆盖」）
对应 §0.5-5 + **P0 验收（合）③**。可选但建议抽验。

1. 把一个会话养到**长上下文**触发 compact（或手动触发压缩 / 跑一轮 review）。

**期望：**
- [ ] compact / review 同样能在**第一个合法对象处收口**、不复读、不挂死；异常时同样落终态 + reason（与 plan 路一致）。

📷 截图点：compact/review 正常完成或可见失败。

## 3. 通过判据 & 收尾动作（关键：按 `AGENTS.md` §二走）

**全部用例通过 = 本设计 §7 的 E/G 拿到真机证据。** 收尾：

1. **回写设计书勾选**（在 PR 或后续提交）：
   - §6.5：勾上 **`P0 验收（合）`**（行 213）、**`P1 验收（合）`**（行 223）、**`T3.3`**（行 235）。
   - §7：勾上 **E**、**G**（A/B/C/D/F 已代码可证）。
   - §11.1 对照表：把 E/G 从 ⏳ 改 ✅，附截图名/SHA。
2. **关 #39（二次修复，必须当场验过）**：
   ```bash
   gh issue view 39                       # 先看状态，确认仍 open + needs-e2e
   gh issue comment 39 --body "已修复 + 当场 E2E 复验通过：<🖱️实测现象 / 截图名 / commit SHA>"
   gh issue edit 39 --remove-label needs-e2e
   gh issue close 39 --reason completed
   ```
   💬 #39 是**二次修复**：没当场点过就**禁止 close**，只能留 open + `needs-e2e` 并评论「已改待当场复验」。
3. **PR #100 关联的 review issue（#97/#98 等 needs-e2e）**：通过则去 `needs-e2e` + 评「E2E 复验通过：<证据>」。

**任一用例不通过：**
- `gh issue reopen 39`（若已关）或保持 open，评论现状 + 证据，**保留 `needs-e2e`**。
- E2E 发现的**新问题** → 按 foreman-e2e skill 新开 `e2e` issue（标题 `[E2E][严重度] …`，正文含 🖱️实测 + 🔎代码 双证据 + 验收勾选框；标题禁用半角双引号，用 `「」`）。
