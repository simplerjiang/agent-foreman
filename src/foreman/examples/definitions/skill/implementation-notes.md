# 技能：实现要点（implementation-notes）

> 「写实现」这一步喂给 AI 的做法手册。**通用脱敏示例**，请按你的真实工程习惯改写。

## 目标
用最小、最清楚的改动让测试变绿，且不给未来埋坑。

## 怎么做好
- **小步前进**：一次只改一件事，能跑、能回退；别一口气堆一大坨难以审阅的改动。
- **复用优先**：动手前先看有没有现成的函数/模块能用，别重复造轮子。
- **顺着现有风格写**：命名、目录结构、错误处理都向周围代码看齐，让新代码像本来就在那儿。
- **边界要稳**：对外部输入做校验，错误路径有明确处置，不靠「正好不会发生」。
- **改完自查**：跑一遍相关测试 + 通读自己的 diff，问一句「这是不是最简单的做法」。

## 别做
- 不偷工：不要用 `TODO`、空实现、注释掉的断言来「让它先过」。
- 不过度工程：不要为「以后可能用到」加一堆没人要的抽象和配置项。
- 不留机密：不要把密钥、密码、内网地址写进代码或日志。
