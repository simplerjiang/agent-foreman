# 技能：先写测试（write-tests）

> 一段「这种活怎么做好」的做法手册，引擎在「写测试」这一步临时喂给 AI。
> 这是一个**通用脱敏示例**——按你团队的真实习惯改写后，存进你自己的库。

## 目标
在动手实现之前，先用测试把「做对了长什么样」钉死。

## 怎么做好
- **先覆盖主路径（happy path）**：最常见、最该工作的那条流程先有一个能跑的断言。
- **再补关键边界**：空输入、超长输入、非法值、并发/重复调用等容易翻车的角落各来一例。
- **一个测试只验一件事**：断言聚焦，失败时一眼看出是哪儿坏了。
- **可读 > 取巧**：测试是活文档，命名说清「输入什么→期望什么」，别为省行数牺牲可读性。
- **不依赖外部网络/真实凭据**：用注入/mock 把外部边界挡住，测试要能离线、可重复地跑。

## 别做
- 不要写「永远绿」的空测试凑数。
- 不要把实现细节焊死进断言（实现一改测试全红，是脆弱测试）。
- 不要在测试里留真实密钥、token、内部地址。
